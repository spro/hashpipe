1 First-class functions: lambdas + def + alias rework

Touches: grammar.peg, src/pipeline.ts, src/builtins/core.ts, src/builtins/collections.ts, src/builtins/environment.ts  ·  SHIPPED 2026-06-11 — tests in tests/lambda.test.ts (42 cases), docs in Syntax.md § Expressions + § Functions. Shipped with three syntax revisions over this plan: lambdas close with } not |}; infix operators landed alongside (+ - * / % and < > <= >= == != with precedence and parens), giving filter real comparison predicates; and Ruby-style params were replaced by a named input before the first pipe (def human-years { $n | $n * 7 }) — the name binds the first argument if given, else the piped input, so human-years 6 and 6 | human-years both work, and the required $ sigil keeps bare-word bodies like {| trim | upper } unambiguous. Notes: sortBy/groupBy accept lambdas only (strings stay key names); the legacy JS-string filter form is retained without a deprecation warning for now.

Problem

Every argument is evaluated before the command runs ($() sub-pipes are call-by-value), so no command can receive an unevaluated pipeline. Control flow can't be written in the language: if evaluates both branches, and filter escapes to raw JS via new Function (collections.ts:233–234). The only abstraction mechanism, alias, is a grammar special case — the body is captured as raw text [^;]+ (grammar.peg:150) so it can't contain ;, runPipeline carries a matching magic-string check (pipeline.ts:278), and through() appends call-site args only to the first command, so aliases are effectively zero-arg. The only path to a parameterized command today is a TypeScript module.

Design

One mechanism, two surfaces. A function value — an unevaluated pipeline with optional named params and a captured scope — is the runtime primitive. It can be created two ways:

  • Lambda literal {| ... |} for inline one-offs (predicates, comparators) — unambiguous today: {| fails the current obj_cmd rule, so nothing breaks.
  • def builtin for named, reusable commands: def name $lambda — a thin wrapper that stores the same value in the scope's fns, making it callable like any command.

Higher-order builtins (filter, map, reduce, if, sortBy, …) accept any of: an inline lambda, a $var holding one, or a bare command name resolved through ctx.get("fns") (shell-style, like xargs cmd) — so naming-first and inline-first workflows both compose.

Before — today
# predicates escape to raw JavaScript:
#| $people | filter 'i.age > 30'

# $() is eager — both branches always run:
#| if $(test) $(then-cmd) $(else-cmd)

# no way to store or pass a pipeline;
# $() evaluates immediately:
#| set double $( * 2 )   # stores a value,
                         # not a command
After — proposed
# inline one-off, no naming required:
#| $people | filter {| @ age | > 30 |}

# branches are lazy; untaken never runs:
#| if $(test) {| then-cmd |} {| else-cmd |}

# lambdas are values — store, pass, call:
#| set double {| * 2 |}
#| [1, 2, 3] | map $double
[ 2, 4, 6 ]
Before — alias today
# aliases are zero-arg; call-site args are
# blindly appended to the FIRST command:
#| alias sevens = * 7
#| 6 | sevens
42        # works only by accident of position

# body cannot contain ';' — parse stops:
#| alias setup = use http; use files   # ✗

# no named parameters at all:
#| alias greet = echo Hello ???        # ✗
After — def
# real named parameters:
#| def human-years {|n| $n | * 7 |}
#| human-years 6
42

# multi-pipeline bodies are fine:
#| def setup {| use http; use files |}

# args usable anywhere in the body;
# defined names work as bare references:
#| def adult {| @ age | > 30 |}
#| $people | filter adult

Plan

  1. Grammar: add lambda = "{|" space params:lambda_params? space body:pipelines space "|}" where lambda_params = (letter_word space)+ "|". Returns {fn: body, params: [...]}. Add lambda to cmd_phrase and start_cmd. Stash the source text on the node via PEG's text() for pretty-printing.
  2. Runtime value: in replaceArg (pipeline.ts:282), when an arg has .fn, do not evaluate — produce a closure object {__lambda__: true, tokens, params, ctx} capturing the defining scope. Add an isLambda() guard in utils/lang.
  3. Invocation: add callLambda(lam, inp, args, cb) in pipeline.ts — creates a subScope of the captured ctx, binds params[i] → args[i] as vars (plus $args for the rest), then runPipelines(lam.tokens, inp, scope, cb).
  4. def builtin (environment.ts): def name $lambda → ctx.topScope().set("fns", name, lambdaAsFn) where lambdaAsFn delegates to callLambda. ~10 lines.
  5. doCmd integration: if the resolved fn from the scope chain is a lambda value, route through callLambda. Add a call builtin (call $f a b) for explicit invocation of a value.
  6. Function-name references: add a resolveCallable(arg, ctx) helper — lambda value passes through; bare string tries ctx.get("fns", arg); used by every higher-order builtin so filter adult and filter {| ... |} both work.
  7. Reimplement alias: keep the alias name = body grammar sugar for back-compat, but desugar to a def whose body is parsed tokens (not raw text) with trailing $args appended to the first command — preserving today's behavior while gaining multi-pipeline bodies and shadowability. Relax the [^;]+ capture to a real pipelines rule, and delete the cmd_args[0] === "alias" magic string in parseArgs (pipeline.ts:278).
  8. Retrofit builtins: if cond $then $else (lazy branches), filter (drop the new Function path, keep the old string form temporarily behind a deprecation warning), new map, reduce, each; sortBy/groupBy accept callables.
  9. Tests: parse round-trips; closure capture (lambda reads a var from defining scope after scope exits); param binding; lambdas stored in vars, passed through ||, nested in @ sub-expressions; bare-name resolution in filter; if/filter laziness (side-effect counter via inc proves the untaken branch never ran); existing alias tests pass unchanged (back-compat gate); alias bodies containing ;; aliases persistence round-trip.

Risks

  • Closure ctx capture keeps scopes alive — fine for a REPL, but document that lambdas hold references.
  • prettyPrint must render lambdas as their source text ({| … |}), not token trees.
  • Serialization: lambdas aren't JSON — stringify should emit the source string (chosen over erroring).
  • Bare-name resolution in higher-order builtins introduces string-vs-name ambiguity — manageable per command, but document it ("filter's argument is a lambda or the name of a command").

Decisions for Sean

  • Lambda literal syntax: {| ... |} (proposed) vs alternatives — settle before step 1.
  • Optional extra sugar: a def name params... = pipelines grammar rule (sister to setting/aliasing) so simple defs don't need braces. Purely additive; can wait for usage feedback.
  • Deprecation horizon for the raw-text alias x = ... sugar. Recommend keeping it (REPL-ergonomic) but documenting def as the primary form.

2 Pipe-operator registry + error pipe |?

Touches: grammar.peg, src/pipeline.ts  ·  Effort: MEDIUM

Problem

A new pipe type touches four grammar rules (token, Xpiped_cmd, ordering-sensitive cmd_section alternation) plus the hardcoded cmd_type if/else chain in runPipeline (pipeline.ts:370–419). Separately, the language has no error channel — any failed stage aborts the pipeline.

Plan

  1. Grammar — one generic rule replacing pipe/ppipe/spipe + the three *piped_cmd rules:
    Before — grammar.peg today
    pipe  = space "|" space
ppipe = space "||" space
spipe = space "|=" space

piped_cmd  = pipe cmd_at:cmd_at
  { cmd_at.type = 'pipe';  return cmd_at; }
ppiped_cmd = ppipe cmd_at:cmd_at
  { cmd_at.type = 'ppipe'; return cmd_at; }
spiped_cmd = spipe cmd_at:cmd_at
  { cmd_at.type = 'spipe'; return cmd_at; }

// order-sensitive: || must precede |
cmd_section = start_cmd / ppiped_cmd
            / spiped_cmd / piped_cmd
    After — proposed
    pipe_op = space "|" suffix:[|=?&^~]? space
  { return "|" + (suffix || ""); }

piped_cmd = op:pipe_op cmd_at:cmd_at
  { cmd_at.type = op; return cmd_at; }

cmd_section = start_cmd / piped_cmd
    PEG longest-match handles || vs | naturally; the ordering hazard in cmd_section disappears. Token types become the operator strings themselves ("|", "||", "|=") — a small mechanical migration from "pipe"/"ppipe"/"spipe".
  2. Runtime registry: 
    type PipeHandler = (cmd_args: any[], inp: any, ctx: Scope,
                    runStage: (args: any[], inp: any, cb: Callback) => void,
                    cb: Callback) => void

const pipeHandlers: Record<string, PipeHandler> = {
  "|":  (args, inp, ctx, run, cb) => run(args, inp, cb),
  "||": (args, inp, ctx, run, cb) => async.map(inp, (i, c) => run(args, i, c), cb),
  "|=": (args, inp, ctx, run, cb) => async.mapSeries(inp, (i, c) => run(args, i, c), cb),
}
export function registerPipeOperator(op: string, h: PipeHandler) { ... }
    runPipeline's if/else chain collapses to a registry lookup; runStage closes over parseArgs + doCmd so handlers stay simple.
  3. Error routing: change the inter-stage continuation (pipeline.ts:360) so an error doesn't abort blindly: if err and the next token's type is "|?", invoke that stage with the error as input and clear the error; otherwise propagate as today. Then register "|?" as a normal handler. This is the proof the registry works:
    Before — today
    # any failed stage aborts the whole
# pipeline; no way to catch or recover:
#| http.get $url | @ items
Error executing pipeline: ECONNREFUSED

# only workaround is handling errors
# inside a TypeScript module
    After — proposed
    # |? catches the upstream error ($! = err):
#| http.get $url |? echo Request failed: $!
'Request failed: ECONNREFUSED'

# or recover with a fallback value:
#| http.get $url | @ items |? null
null

# skipped entirely when nothing failed
  4. Module-provided operators (stretch): recognize an __operators__ export in Pipeline.use and register through the same API. Cheap once the registry exists.
  5. Tests: all existing pipe tests green after token-type rename; |? catches an error mid-pipeline and downstream stages continue; |? skipped entirely when no error; unknown operator char → parse error (not silent |).

Risks

  • ok_char includes = ? ^ — confirm operator suffixes can't be eaten by a preceding cmd_word. Safe because pipe_op requires the leading | which is not in ok_char, but add a regression test for echo a=b | echo c?d.
  • Error-routing changes the abort path — audit tests-hp/ scripts that rely on hard failure.

3 Command lookup order — builtins last

Touches: src/pipeline.ts (doCmd), src/builtins/debug.ts  ·  Effort: SMALL

Problem

doCmd (pipeline.ts:434) checks builtins[cmd] before the scope chain, so modules, aliases, and user defs can never override a builtin — the root cause behind the "ls is no longer clobbered" class of collisions.

Plan

  1. Flip the lookup: ctx.get("fns", cmd) first, builtins[cmd] as fallback. Two-line change.
  2. Preferred end-state (slightly bigger): seed the Pipeline root scope's fns with all builtins at construction, delete the direct builtins import from doCmd, and let normal scope-chain resolution do everything. One lookup path; shadowing falls out of scoping for free.
  3. Escape hatch: add a builtin command (builtin echo hi) that bypasses the scope chain, and a commands/which introspection builtin showing where a name resolves (builtin / module / alias / def) — mirrors bash type.
  4. Tests: alias shadows builtin and builtin x still reaches the original; use-loaded fn shadows builtin; unshadowed builtins resolve identically to today (run full suite).

Risk

Behavioral: a module that accidentally exports get-style generic names now wins over builtins. That's the point, but call it out in the changelog and have use print shadow warnings (pairs with #4's collision reporting).

4 User module search path

Touches: src/pipeline.ts (Pipeline.use)  ·  Effort: SMALL

Problem

use resolves bare names to ./modules/ inside the installed package (pipeline.ts:139–143). Users can't add a module without publishing it into hashpipe's own lib/.

Plan

  1. Resolution order for use name:
    1. Explicit path (./x, /x, ~/x) → resolve against process.cwd(), expand ~
    2. Each dir in HASHPIPE_PATH (colon-separated)
    3. ~/.hashpipe/modules/
    4. Bundled ./modules/ (today's behavior)
    5. npm: hashpipe-<name> then <name>, via module.createRequire(process.cwd() + "/") so local node_modules works
  2. Implementation: extract a resolveModule(name): string helper with the chain above; first hit wins; aggregate the attempted paths into the error message on total miss (bash-style "command not found" ergonomics).
  3. Collision reporting: registerNamespace currently overwrites silently — collect names that already existed and include "shadowing: x, y" in the use result message.
  4. Tests: temp-dir fixture module loaded via each tier; precedence order (HASHPIPE_PATH beats bundled); miss lists attempted locations.

5 Grammar paper cuts

Touches: grammar.peg, src/pipeline.ts (parseArgs)  ·  Effort: MEDIUM (four independent small fixes)

5a — Comments

# is in ok_char (URLs contain fragments), so comments must be position-aware, not lexical. Plan: introduce a statement-level rule — comment = space "#" [^\n;]* — allowed wherever a pipeline can appear in the pipelines list, and treat newline as a pipeline separator alongside ; in file/--run mode. # mid-word stays a literal character.

Before — script.hp today
# this line is a parse error, not a comment
use http
http.get http://x.io/page#section
^ newlines are whitespace: these two lines
  collide into one command unless joined
  with ';'
After — proposed
# fetch the section anchor demo
use http
http.get http://x.io/page#section
^ '#' at statement start = comment;
  '#' mid-word still literal (URL intact);
  newline separates pipelines like ';'

Decision needed: newline-as-separator changes script semantics (currently newlines are whitespace) — gate it on execFile only, or accept globally? Recommend globally; multi-line single pipelines keep working via trailing-| continuation.

5b — Generic escaping

Replace the enumerated escape rules (grammar.peg:72–75) with one generic rule (keep \n → newline in quoted literals). Every future metacharacter is automatically escapable.

Before — grammar.peg today
// each escape individually enumerated:
escd_at    = ... "\\@" { return "@"; }
escd_paren = "\\(" { return "("; }
           / "\\)" { return ")"; }
escd_dolla = "\\$"
escd_char  = escd_at / escd_paren
           / escd_dolla

# so only these work; \| \; \# etc. don't:
#| echo a \| b      # ✗ parse error
After — proposed
// one rule covers every metacharacter,
// present and future:
escd_char = "\\" c:. { return c; }

#| echo a \| b
'a | b'
#| echo 100\% \#1
'100% #1'

Test matrix: \@ \( \) \$ \| \; \# \\ in word, single-quoted, and double-quoted positions.

5c — Quoted keys + wildcard in at-expressions

at_word only matches [a-zA-Z0-9_~/-], so JSON keys with spaces/dots are unreachable. Plan: section = at_obj / at_array / at_quoted / at_word where at_quoted reuses quoted_literal and yields the raw string. Add wildcard: @ :* / @ .* where * in accessor() means Object.values(obj).

Before — today
# keys with spaces/dots are unreachable:
#| {"content type": "json"} @ ???      # ✗
#| {"v1.2": "ok"} @ v1.2   # ✗ '.' parses
                           #   as a get step

# no way to take all values of an object:
#| {a: 1, b: 2} @ ???      # ✗ must pipe to
                           #   'values' cmd
After — proposed
# quoted keys reach any JSON key:
#| {"content type": "json"} @ "content type"
'json'
#| {"v1.2": "ok"} @ "v1.2"
'ok'

# wildcard = Object.values, chainable:
#| {a: {n: 1}, b: {n: 2}} @ *:n
[ 1, 2 ]

Defer slices ([0..3] range syntax) — nice-to-have, doesn't block anything.

5d — ${var} + typed interpolation + replace bug

Before — today
# no delimiter form — $foo_bar is
# ambiguous, can't splice $foo + "_bar":
#| echo $file_v2          # reads var
                          # 'file_v2'

# objects stringify when embedded:
#| set u {name: "Al"}
#| list $u
[ '[object Object]' ]     # type lost
After — proposed
# braces disambiguate:
#| echo ${file}_v2
'data.json_v2'

# a lone-var arg passes its value
# through with type intact:
#| set u {name: "Al"}
#| list $u
[ { name: 'Al' } ]
  1. Grammar: var_word = "${" w:letter_word "}" / "$" w:letter_word / "$!" — disambiguates ${foo}_bar.
  2. Typed pass-through: in parseArgs, when an argument is exactly one variable token, substitute the value as-is (object stays object); only fall back to string-splicing when the var is embedded in a larger word. Requires cmd_word to return its parts rather than a pre-joined string — the one structural grammar change here.
  3. Fix the latent bug at pipeline.ts:317: arg.replace(key_str, val) mis-expands values containing $&/$' — use the function form arg.replace(key_str, () => String(val)). Worth shipping immediately regardless of the rest.
  4. Also: drop the dead bool rule (grammar.peg:70) and add a null literal to start_cmd beside bool_word. Leave bare-word args as strings (changing that is a breaking change with no extensibility payoff).
