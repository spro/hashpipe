1 First-class functions: lambdas + def + alias rework

Touches: grammar.peg, src/pipeline.ts, src/builtins/core.ts, src/builtins/collections.ts, src/builtins/environment.ts  ·  SHIPPED 2026-06-11 — tests in tests/lambda.test.ts (50 cases), docs in Syntax.md § Expressions + § Functions. The design below reflects the final shipped syntax, revised twice during review. Notes: sortBy/groupBy accept lambdas only (strings stay key names); the legacy JS-string filter form is retained without a deprecation warning for now.

Problem

Before this work, every argument was evaluated before the command ran ($() sub-pipes were call-by-value), so no command could receive an unevaluated pipeline. Control flow could not be written in the language: if evaluated both branches, and filter escaped to raw JS via new Function (collections.ts:233–234). The only abstraction mechanism, alias, was a grammar special case — the body was captured as raw text [^;]+ (grammar.peg:150) so it could not contain ;, runPipeline carried a matching magic-string check (pipeline.ts:278), and through() appended call-site args only to the first command, so aliases were effectively zero-arg. The only path to a parameterized command was a TypeScript module.

Design

One mechanism, two surfaces. A function value — an unevaluated pipeline with a captured scope and an optionally named input — is the runtime primitive. It can be created two ways:

  • Lambda literal {| ... } for inline one-offs (predicates, comparators): a pipeline in a box. Naming the input extends the same shape — { $n | ... } — with the $ sigil marking the binding, so bare words before a pipe are never captured ({| trim | upper } is always a two-stage body).
  • def builtin for named, reusable commands: def name { $n | ... } — a thin wrapper that stores the same value in the scope's fns, making it callable like any command.

Binding rule (decided): a name binds the first call-site argument if one is given, and the piped input otherwise — so dog-years 6 and 6 | dog-years both work (precedent: set's dual behavior). Extra names bind remaining arguments in order; leftovers land in $args. Two earlier drafts were rejected on the way here: Ruby-style {|n| ...} params (the closing param pipe collides with the pipe operator, so {| trim | upper } silently ate trim as a param name) and the |} closing delimiter (redundant — plain } closes). Infix expressions (+ - * / % and < > <= >= == != with precedence and parens) landed alongside so predicates read naturally.

Higher-order builtins (filter, map, reduce, if, sortBy, …) accept any of: an inline lambda, a $var holding one, or a bare command name resolved through ctx.get("fns") (shell-style, like xargs cmd) — so naming-first and inline-first workflows both compose.

Before — old behavior
# predicates escape to raw JavaScript:
#| $people | filter 'i.age > 30'

# $() is eager — both branches always run:
#| if $(test) $(then-cmd) $(else-cmd)

# no way to store or pass a pipeline;
# $() evaluates immediately:
#| set double $( * 2 )   # stores a value,
                         # not a command
After — shipped
# inline one-off, no naming required:
#| $people | filter {| $(@ age) > 30 }

# branches are lazy; untaken never runs:
#| if $(test) {| then-cmd } {| else-cmd }

# lambdas are values — store, pass, call:
#| set double {| * 2 }
#| [1, 2, 3] | map $double
[ 2, 4, 6 ]
Before — alias behavior
# aliases are zero-arg; call-site args are
# blindly appended to the FIRST command:
#| alias sevens = * 7
#| 6 | sevens
42        # works only by accident of position

# body cannot contain ';' — parse stops:
#| alias setup = use http; use files   # ✗

# no named parameters at all:
#| alias greet = echo Hello ???        # ✗
After — def (shipped)
# named input: binds the argument if
# given, else the piped input:
#| def dog-years { $n | $n * 7 }
#| dog-years 6
42
#| 6 | dog-years
42

# multi-pipeline bodies are fine:
#| def setup {| use http ; use files }

# defined names work as bare references:
#| def adult {| $(@ age) > 30 }
#| $people | filter adult

Shipped Details

  1. Grammar (as shipped): lambda = "{" space params:lambda_params? "|" space body:pipelines space "}" where lambda_params is one or more $-sigiled names before the first pipe. Returns {fn: body, params: [...], src: text()}. lambda added to cmd_phrase and start_cmd.
  2. Runtime value (as shipped): a Lambda class in helpers.ts (the pipeline runner is injected via setLambdaRunner to avoid a circular import); replaceArg constructs it from the grammar token, capturing the defining scope.
  3. Invocation (as shipped): Lambda.call(inp, args, cb) — creates a subScope of the captured ctx, binds each name to its positional argument or, when no argument was given, to the piped input (dual binding), stashes the rest as $args, then runs the body tokens.
  4. def builtin (environment.ts): def name {| ... } → ctx.topScope().set("fns", name, lambda); rejects non-lambda bodies.
  5. doCmd integration: a resolved fn that is a Lambda routes through .call. A call builtin (call $f a b) invokes a value explicitly.
  6. Function-name references: add a resolveCallable(arg, ctx) helper — lambda value passes through; bare string tries ctx.get("fns", arg); used by every higher-order builtin so filter adult and filter {| ... |} both work.
  7. alias compatibility: alias name = body remains a single-pipeline shortcut that stops at ;. The body is parsed into an unevaluated lambda token so definition-time substitution does not happen, while call-site args are still appended to the first command to preserve classic alias behavior. Multi-pipeline and named-parameter workflows use def.
  8. Retrofit builtins: if cond $then $else uses lazy lambda branches; filter, map, reduce, each, sortBy, and groupBy accept callables.
  9. Tests: parse round-trips; closure capture; param binding; lambdas stored in vars, passed through ||, nested in @ sub-expressions; bare-name resolution in higher-order commands; lazy branch behavior; alias script reporting; alias call-site args; alias bodies stopping at semicolon; and aliases persistence.

Risks

  • Closure ctx capture keeps scopes alive — fine for a REPL, but document that lambdas hold references.
  • prettyPrint must render lambdas as their source text ({| … }), not token trees.
  • Serialization: lambdas aren't JSON — stringify should emit the source string (chosen over erroring).
  • Bare-name resolution in higher-order builtins introduces string-vs-name ambiguity — manageable per command, but document it ("filter's argument is a lambda or the name of a command").

Decisions — resolved

  • Lambda literal syntax: settled on {| body } for pipe-through and { $n | body } for named input, after three iterations — the originally proposed {| ... |} closer and Ruby-style {|n| ...} params were both rejected (see Design for why). Guiding principles: the $ sigil makes bindings reader-decidable, parse errors stay loud, and { key: ... } remains unmistakably JSON data.
  • Brace-free def name params... = pipelines sugar: deferred; revisit with usage feedback.
  • alias: kept as the lighter cousin (single-pipeline body stopping at ;, call-site args appended to the first command, caller's scope), with def documented as the primary form. No deprecation.

2 Pipe-operator registry + error pipe |?

Touches: grammar.peg, src/pipeline.ts  ·  SHIPPED 2026-06-12 — tests in tests/pipes.test.ts cover normal, parallel, series, and error pipes. The shipped registry uses operator-string token types (|, ||, |=, |?); module-provided operators remain a stretch item.

Problem

A new pipe type touches four grammar rules (token, Xpiped_cmd, ordering-sensitive cmd_section alternation) plus the hardcoded cmd_type if/else chain in runPipeline (pipeline.ts:370–419). Separately, the language has no error channel — any failed stage aborts the pipeline.

Shipped Details

  1. Grammar — one generic rule replacing pipe/ppipe/spipe + the three *piped_cmd rules:
    Before — old grammar.peg
    pipe  = space "|" space
ppipe = space "||" space
spipe = space "|=" space

piped_cmd  = pipe cmd_at:cmd_at
  { cmd_at.type = 'pipe';  return cmd_at; }
ppiped_cmd = ppipe cmd_at:cmd_at
  { cmd_at.type = 'ppipe'; return cmd_at; }
spiped_cmd = spipe cmd_at:cmd_at
  { cmd_at.type = 'spipe'; return cmd_at; }

// order-sensitive: || must precede |
cmd_section = start_cmd / ppiped_cmd
            / spiped_cmd / piped_cmd
    After — shipped
    pipe_op = space "|" suffix:[|=?&^~]? space
  { return "|" + (suffix || ""); }

piped_cmd = op:pipe_op cmd_at:cmd_at
  { cmd_at.type = op; return cmd_at; }

cmd_section = start_cmd / piped_cmd
    PEG longest-match handles || vs | naturally; the ordering hazard in cmd_section disappears. Token types become the operator strings themselves ("|", "||", "|=") — a small mechanical migration from "pipe"/"ppipe"/"spipe".
  2. Runtime registry: 
    type PipeHandler = (
  inp: any,
  ctx: Scope,
  runStage: (inp: any) => Promise<any>,
) => Promise<any>

const pipeHandlers: Record<string, PipeHandler> = {
  "|":  (inp, ctx, run) => run(inp),
  "|?": (inp, ctx, run) => run(inp),
  "||": (inp, ctx, run) => Promise.all(inp.map(run)),
  "|=": async (inp, ctx, run) => { ... },
}
export function registerPipeOperator(op: string, h: PipeHandler) { ... }
    runPipeline's if/else chain collapses to a registry lookup; runStage closes over argument parsing and command execution so handlers stay simple.
  3. Error routing: change the inter-stage continuation (pipeline.ts:360) so an error doesn't abort blindly: if err and the next token's type is "|?", invoke that stage with the error as input and clear the error; otherwise propagate normally. Then register "|?" as a normal handler. This is the proof the registry works:
    Before — old behavior
    # any failed stage aborts the whole
# pipeline; no way to catch or recover:
#| http.get $url | @ items
Error executing pipeline: ECONNREFUSED

# only workaround is handling errors
# inside a TypeScript module
    After — shipped
    # |? catches the upstream error ($! = err):
#| http.get $url |? echo Request failed: $!
'Request failed: ECONNREFUSED'

# or recover with a fallback value:
#| http.get $url | @ items |? null
null

# skipped entirely when nothing failed
  4. Module-provided operators (stretch): recognize an __operators__ export in Pipeline.use and register through the same API. Cheap once the registry exists.
  5. Tests: all existing pipe tests green after token-type rename; |? catches an error mid-pipeline and downstream stages continue; |? skipped entirely when no error; unknown operator char → parse error (not silent |).

Risks

  • ok_char includes = ? ^ — confirm operator suffixes can't be eaten by a preceding cmd_word. Safe because pipe_op requires the leading | which is not in ok_char, but add a regression test for echo a=b | echo c?d.
  • Error-routing changes the abort path — audit tests-hp/ scripts that rely on hard failure.

3 Command lookup order — builtins last

Touches: src/pipeline.ts (doCmd), src/builtins/meta.ts  ·  SHIPPED 2026-06-12 — command lookup now checks the active scope before builtins, and meta helpers expose resolution behavior.

Problem

doCmd (pipeline.ts:434) checks builtins[cmd] before the scope chain, so modules, aliases, and user defs can never override a builtin — the root cause behind the "ls is no longer clobbered" class of collisions.

Shipped Details

  1. Flip the lookup: ctx.get("fns", cmd) first, builtins[cmd] as fallback. Two-line change.
  2. Escape hatch: the builtin command (builtin echo hi) bypasses the scope chain, and which reports where a name resolves (builtin / module / alias / def) — mirrors bash type.
  3. Collision reporting: module registration reports shadowed names through the use result.
  4. Tests: alias shadows builtin and builtin x still reaches the original; use-loaded functions shadow builtins; unshadowed builtins resolve identically to before.

Risk

Behavioral: a module that accidentally exports get-style generic names now wins over builtins. That's the point, and use reports shadow warnings so collisions are visible.

4 Grammar paper cuts

Touches: grammar.peg, src/pipeline.ts (parseArgs)  ·  SHIPPED 2026-06-12 — tests in tests/grammar.test.ts, docs in Syntax.md (§ Statements and comments, § Variables, § At-expressions). The shipped set includes newline-as-separator globally, # comments at statement position, one generic escape rule (\$ keeps its backslash for parseArgs), quoted keys, * wildcard and range slices in at-expressions, ${var} splicing, typed lone-var pass-through, the $& replace bug fix, a null literal, and removal of the dead bool rule.

4a — Comments

# is in ok_char (URLs contain fragments), so comments must be position-aware, not lexical. Plan: introduce a statement-level rule — comment = space "#" [^\n;]* — allowed wherever a pipeline can appear in the pipelines list, and treat newline as a pipeline separator alongside ; in file/--run mode. # mid-word stays a literal character.

Before — old script.hp
# this line is a parse error, not a comment
use http
http.get http://x.io/page#section
^ newlines are whitespace: these two lines
  collide into one command unless joined
  with ';'
After — shipped
# fetch the section anchor demo
use http
http.get http://x.io/page#section
^ '#' at statement start = comment;
  '#' mid-word still literal (URL intact);
  newline separates pipelines like ';'

Decision (resolved): newline-as-separator was accepted globally, as recommended. Multi-line pipelines keep working via leading- or trailing-pipe continuation.

4b — Generic escaping

Replace the enumerated escape rules (grammar.peg:72–75) with one generic rule (keep \n → newline in quoted literals). Every future metacharacter is automatically escapable.

Before — old grammar.peg
// each escape individually enumerated:
escd_at    = ... "\\@" { return "@"; }
escd_paren = "\\(" { return "("; }
           / "\\)" { return ")"; }
escd_dolla = "\\$"
escd_char  = escd_at / escd_paren
           / escd_dolla

# so only these work; \| \; \# etc. don't:
#| echo a \| b      # ✗ parse error
After — shipped
// one rule covers every metacharacter,
// present and future:
escd_char = "\\" c:. { return c; }

#| echo a \| b
'a | b'
#| echo 100\% \#1
'100% #1'

Test matrix: \@ \( \) \$ \| \; \# \\ in word, single-quoted, and double-quoted positions.

4c — Quoted keys + wildcard in at-expressions

at_word only matches [a-zA-Z0-9_~/-], so JSON keys with spaces/dots are unreachable. Plan: section = at_obj / at_array / at_quoted / at_word where at_quoted reuses quoted_literal and yields the raw string. Add wildcard: @ :* / @ .* where * in accessor() means Object.values(obj).

Before — old behavior
# keys with spaces/dots are unreachable:
#| {"content type": "json"} @ ???      # ✗
#| {"v1.2": "ok"} @ v1.2   # ✗ '.' parses
                           #   as a get step

# no way to take all values of an object:
#| {a: 1, b: 2} @ ???      # ✗ must pipe to
                           #   'values' cmd
After — shipped
# quoted keys reach any JSON key:
#| {"content type": "json"} @ "content type"
'json'
#| {"v1.2": "ok"} @ "v1.2"
'ok'

# wildcard = Object.values, chainable:
#| {a: {n: 1}, b: {n: 2}} @ *:n
[ 1, 2 ]

Range slices shipped too, using @ 1..3, @ ..3, @ 2.., and negative bounds such as @ -3..-1.

4d — ${var} + typed interpolation + replace bug

Before — old behavior
# no delimiter form — $foo_bar is
# ambiguous, can't splice $foo + "_bar":
#| echo $file_v2          # reads var
                          # 'file_v2'

# objects stringify when embedded:
#| set u {name: "Al"}
#| list $u
[ '[object Object]' ]     # type lost
After — shipped
# braces disambiguate:
#| echo ${file}_v2
'data.json_v2'

# a lone-var arg passes its value
# through with type intact:
#| set u {name: "Al"}
#| list $u
[ { name: 'Al' } ]
  1. Grammar: var_word = "${" w:letter_word "}" / "$" w:letter_word / "$!" — disambiguates ${foo}_bar.
  2. Typed pass-through: in parseArgs, when an argument is exactly one variable token, substitute the value as-is (object stays object); only fall back to string-splicing when the var is embedded in a larger word. Requires cmd_word to return its parts rather than a pre-joined string — the one structural grammar change here.
  3. Fix the latent bug at pipeline.ts:317: arg.replace(key_str, val) mis-expands values containing $&/$' — use the function form arg.replace(key_str, () => String(val)). Worth shipping immediately regardless of the rest.
  4. Also: drop the dead bool rule (grammar.peg:70) and add a null literal to start_cmd beside bool_word. Leave bare-word args as strings (changing that is a breaking change with no extensibility payoff).

6 User module search path

Touches: src/module-loader.ts (new), src/pipeline.ts (Pipeline.use), src/builtins/environment.ts  ·  SHIPPED 2026-06-12 — tests in tests/modules.test.ts, docs in Modules.md § Where modules live. Shipped with one design change from review (per Sean): the npm tier tries only hashpipe-<name>, never the generic package name. Resolution order otherwise as planned (explicit paths → HASHPIPE_PATH → ~/.hashpipe/modules → bundled → npm), a total miss lists every attempted location, and use reports "(shadowing: ...)" when a load overwrites existing commands.

Problem

use resolves bare names to ./modules/ inside the installed package (pipeline.ts:139–143). Users can't add a module without publishing it into hashpipe's own lib/.

Shipped Details

  1. Resolution order for use name:
    1. Explicit path (./x, /x, ~/x) → resolve against process.cwd(), expand ~
    2. Each dir in HASHPIPE_PATH (colon-separated)
    3. ~/.hashpipe/modules/
    4. Bundled package modules
    5. npm: hashpipe-<name> only, via module.createRequire(process.cwd() + "/") so local node_modules works
  2. Implementation: extract a resolveModule(name): string helper with the chain above; first hit wins; aggregate the attempted paths into the error message on total miss (bash-style "command not found" ergonomics).
  3. Collision reporting: registerNamespace currently overwrites silently — collect names that already existed and include "shadowing: x, y" in the use result message.
  4. Tests: temp-dir fixture module loaded via each tier; precedence order (HASHPIPE_PATH beats bundled); miss lists attempted locations.
